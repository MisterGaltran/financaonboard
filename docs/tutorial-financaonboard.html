01Atualizar o app

A regra: push pra main no GitHub. Frontend rebuilda sozinho. Backend você dispara manualmente.

Fluxo do FRONTEND

Mexeu em qualquer arquivo dentro de frontend/? Só precisa:

# no PowerShell do notebook
cd C:\financaonboard
git add frontend/
git commit -m "ajuste no header"
git push
O Cloudflare Pages tem webhook no GitHub. Detecta o push em ~10s, faz npm run build em frontend/, sobe pra CDN. Total: ~1-2 min. Acompanhe em dash.cloudflare.com → Workers & Pages → financaonboard → Deployments.

Fluxo do BACKEND

Mexeu em algo dentro de backend/? Após push, conecta no Yellow e roda o script de deploy:

# no SSH do Yellow
bash /mnt/projects/financaonboard/infra/deploy.sh

O que esse script faz, em ordem:

  1. git pull --ff-only — atualiza o repo local com o main remoto.
  2. npm ci --omit=dev — instala deps de produção exatas (lockfile).
  3. sudo systemctl restart financaonboard-api — reinicia o serviço.
  4. Espera 3s e mostra status + últimas 20 linhas do log.

Mudou variável de ambiente do backend?

Edita o .env direto e reinicia:

sudo nano /mnt/projects/financaonboard/backend/.env
sudo systemctl restart financaonboard-api

Mudou variável do frontend (VITE_*)?

No painel do Cloudflare Pages: Settings → Variables and Secrets → Production. Salva e clica em Retry deployment no último deploy. Vite injeta envs em build-time, então o rebuild é necessário.

Adicionou nova API key no .env

  1. No notebook, edita o backend/.env local pra desenvolvimento.
  2. Faz scp backend/.env yellow-vpn:/mnt/projects/financaonboard/backend/.env.
  3. No Yellow: chmod 600 /mnt/projects/financaonboard/backend/.env.
  4. sudo systemctl restart financaonboard-api.
Cuidado com o .env: nunca commite no Git. O backend/.gitignore já cobre, mas confirme com git status antes de qualquer git add.

02Monitorar e diagnosticar

Comandos do dia-a-dia. Tudo do SSH do Yellow.

Status rápido

systemctl is-active financaonboard-api tailscaled
tailscale funnel status
# saída esperada: active / active + URL pública listada

Logs ao vivo do backend

journalctl -u financaonboard-api -f

Eventos esperados em condições normais:

  • HTTP + Socket.io listening on :4000 ao subir
  • RSS <Feed>: N new a cada 90s
  • Currency quotes refreshed: 5 pairs a cada 15s
  • BR quotes refreshed (Yahoo): N symbols a cada 15s
  • WARN Polygon WS closed (1006) ocasional — reconecta automaticamente

Logs do Tailscale

journalctl -u tailscaled -f

Em condições normais é bem verbose mas tudo INFO. Procure por "ERR" se desconfiar de algo.

Health check externo

curl https://yellow-vpn.tailf47dec.ts.net/health
# {"ok":true,"uptime":...,"providers":{"finnhub":true,"eodhd":true,"polygon":true,"brapi":true}}

Se algum provider vier false: a chave dele está faltando ou inválida no .env.

Saúde da máquina

uptime              # carga e tempo ligado
free -h             # memória
df -h /mnt/projects # disco
htop                # interativo
sudo nvme smart-log /dev/nvme0n1 | head -10  # saúde do SSD
vcgencmd measure_temp 2>/dev/null            # temperatura

Tráfego do app

# conexões ativas no backend (porta 4000, deve vir só de localhost / Tailscale Funnel)
sudo ss -tan state established 'sport = :4000' | head -20

# status atual do Funnel
tailscale funnel status

Erro recente?

# últimos 50 erros do journal
sudo journalctl -p err -n 50 --no-pager

# todas as units do systemd que falharam
systemctl --failed
Dica: se quiser saber se alguém reiniciou o serviço, journalctl -u financaonboard-api --no-pager | grep "Started" mostra todas as inicializações com timestamp.

04Troubleshooting

Problemas comuns e o que fazer.

Site não carrega: financaonboard.pages.dev fora do ar

É problema do Cloudflare (extremamente raro) ou do build do frontend.

# Status do Cloudflare
https://www.cloudflarestatus.com/

# Último deploy do Pages
dash.cloudflare.com → Pages → financaonboard → Deployments

Se um deploy quebrou: clica em View build log para ver o erro. Geralmente é erro de build no frontend (typo, dependência faltando).

App carrega mas indicador de conexão fica vermelho

Backend ou Tailscale fora do ar. Diagnostica do Yellow:

systemctl is-active financaonboard-api tailscaled
tailscale funnel status   # tem que listar a URL com proxy
# se algum vier "failed" ou "inactive":
sudo systemctl restart financaonboard-api tailscaled
journalctl -u financaonboard-api -n 50 --no-pager

Health check da raiz funciona mas WebSocket não conecta

Quase sempre CORS. O backend só aceita origins de FRONTEND_URL:

grep FRONTEND_URL /mnt/projects/financaonboard/backend/.env
# tem que ser https://financaonboard.pages.dev
# se mudou de domínio no Pages, atualize aqui e:
sudo systemctl restart financaonboard-api

TLS expirou (cert vencido)

Tailscale renova automaticamente. Se algo der errado:

sudo journalctl -u tailscaled --since "1 day ago" | grep -iE "cert|acme|error"
sudo tailscale cert yellow-vpn.tailf47dec.ts.net   # força renovação
sudo systemctl restart tailscaled

Yellow caiu / queda de luz

Tudo volta sozinho quando a luz retornar:

  • systemd inicia caddy, financaonboard-api, wg-quick@wg0.
  • cron retoma DDNS e backups.
  • Backend é stateless — recupera estado das APIs externas em segundos.

Se algo não subiu: systemctl --failed e vê o que falhou.

IP da Oi mudou e a VPN parou

O DDNS do WireGuard atualiza em até 5 min. Forçar manual:

bash /home/davi/desec/update.sh
tail -3 /home/davi/desec/update.log
# tem que aparecer "good" na resposta da deSEC API

O Tailscale Funnel não depende do IP residencial (é outbound), então a URL pública não muda.

Frontend mostra dados antigos

Cache do navegador. Ctrl+Shift+R faz hard refresh.

Quero rollback pra versão anterior do backend

cd /mnt/projects/financaonboard
git log --oneline -5                    # ver commits recentes
git checkout <commit-hash-anterior>
cd backend && npm ci --omit=dev
sudo systemctl restart financaonboard-api
# quando estabilizar, voltar pra main:
cd /mnt/projects/financaonboard && git checkout main

Quero rollback do frontend

No painel do Cloudflare Pages → Deployments → escolhe um deploy antigo → Rollback to this deployment.

05Adicionar um segundo backend público

Caso queira no futuro hospedar outro app no mesmo Yellow, com HTTPS público.

Visão geral do que muda

AspectoO que fazer
Pasta do projeto/mnt/projects/<novo-app>/
Porta interna3000-3099 (Node), 5000-5099 (Python). Confirme com sudo ss -tulpn | grep LISTEN
systemd unit/etc/systemd/system/<novo-app>.service
Exposição públicaTailscale Funnel via path

Passo a passo

1. Reservar porta, clonar projeto, criar systemd unit — igual ao do financaonboard. Use o template em infra/financaonboard-api.service como base, troque o WorkingDirectory e a porta no .env.

2. Expor via Tailscale Funnel usando paths

Funnel só roteia uma porta como raiz. Pra múltiplos apps, use paths distintos:

# reset config atual
sudo tailscale funnel reset

# expõe app1 em /api/financa e app2 em /api/outro
sudo tailscale serve --bg --https=443 --set-path=/api/financa http://localhost:4000
sudo tailscale serve --bg --https=443 --set-path=/api/outro http://localhost:5000
sudo tailscale funnel --bg 443 on

# valida
tailscale funnel status

Os apps responderão em:

  • https://yellow-vpn.tailf47dec.ts.net/api/financa/...
  • https://yellow-vpn.tailf47dec.ts.net/api/outro/...

Atualize o VITE_BACKEND_URL de cada frontend para incluir o path correspondente.

Alternativa: rodar um proxy local (Caddy ou Nginx escutando só em localhost) que multiplexa baseado em headers e expor só esse proxy via Funnel.

Atenção ao orçamento de RAM: Yellow tem 4GB total. Cada app novo come 256-512 MB. Já tem ~1.8 GB usados (sistema + financaonboard). Acima de 3 apps simultâneos, considera rodar Postgres compartilhado em vez de SQLite por app.
